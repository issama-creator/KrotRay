Задумка на потом (ключ доступа после оплаты в Telegram-боте)
==========================================================

Формат ключа (пример идеи)
--------------------------
  krot_<token>/<telegram_id>

Пример: пользователь после оплаты получает ключ, в конце — его Telegram user id.
Сохраняется в БД в связке telegram_id + device_stable_id.

Клиент
------
Пользователь вставляет ключ в приложение → запрос на бэкенд → находим ключ,
парсим суффикс telegram_id, проверяем подписку → если всё ок — выдаём сервера
(GET /servers как сейчас) до просрочки подписки.

Реализация на бэкенде (когда будем делать)
------------------------------------------
- Парсинг: один раз описать regex или split по последнему "/" или фиксированному шаблону.
- Опереться на существующие access_keys и GET /servers?key=... / лимиты устройств.
- Новый префикс krot_ можно добавить без ломания старых ключей.

На что обратить внимание
-------------------------
- Уникальность ключа в БД.
- Срок действия ключа / подписки.
- Отзыв ключа (админка / ручной ban).
- Суффикс telegram_id должен совпадать с тем, кому ключ выдали в боте (или
  проверяться против записи в БД после выдачи — не доверять только строке ключа).

Примечание
----------
Сейчас только спецификация; код под это пока не пишем.
